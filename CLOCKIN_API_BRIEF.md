## Authentication

There are **two bearer tokens**. Which one to send depends on the URL prefix. The header format is always:

```
Authorization: Bearer <token>
```

Tokens are opaque strings shaped like `<numeric_id>|<random>` (e.g. `1108226|DuqsHkOH7r1...`). Don't parse them.

### Obtaining tokens

`POST /v2/auth/user/login` with `{email, password}` (`Credentials` schema). Response for `type=personal`:

```json
{ "data": {
    "auth_token":   "<id>|<random>",
    "user_token":   "<id>|<random>",   // for /correction, /timesheet, /admin/*, ...
    "device_token": "<id>|<random>",   // for /events, /projects, /workdays, ...
    "type":         "personal"
}}
```

`auth_token` equals `user_token` for personal accounts; you only need `user_token` and `device_token`. Team-device flows (`/auth/code/login`, `/auth/user/loginWithPin`) return different subsets — see `AuthenticationService.ts`.

### Token routing rules

From `src/composables/OpenApiComposable.ts`:

```ts
const noAuthEndpoints = [
  '/device/upgrade/appv1/appv2',
  '/auth/login',
  '/auth/code/login',
  '/auth/user/login',
  '/auth/user/resetPassword',
  '/registration/company',
  '/registration/v2/company',
]

const userEndpointPrefixes = [
  '/absence', '/company-absence-types', '/correction', '/timesheet',
  '/auth/user/logout', '/auth/user/officeSessionUrl',
  '/auth/user/authorizeQrLogin', '/survey', '/possibleAbsenceSubstitutes',
  '/travelLog/showMany', '/admin/nfc-tags', '/shift-plan', '/admin/',
  '/app-onboarding-tours',
]

// Resolution:
//   exact match in noAuthEndpoints       → no Authorization header
//   startsWith any userEndpointPrefixes  → Bearer user_token
//   else                                 → Bearer device_token
```

Note: `/timesheet` is a prefix, so `/timesheets/totals` and `/timesheets/{start}/{end}` both route to `user_token`. `/workdays`, `/projects`, `/events/storeMany`, `/notifications`, etc. fall through to `device_token`.

### Required header on every authenticated request

The app sends a `device-information` header with a stringified JSON object:

```
device-information: {"app_build":"42","app_version":"2.x","device_model":"...", "operating_system":"...","os_version":"..."}
```

For an MCP, send a constant payload identifying this client:

```ts
'device-information': JSON.stringify({
  app_build: '0',
  app_version: pkg.version,
  device_model: 'mcp',
  operating_system: 'mcp',
  os_version: '1',
})
```

This is required — missing the header may cause 4xx on some endpoints.

## The event model — clock in / clock out / switch task

There is **no** dedicated clock-in/clock-out endpoint. All state changes are **events** posted in batches to:

```
POST /v2/events/storeMany
[
  {
    "uuid":              "<uuidv4>",
    "occured_at":        "2026-05-28T06:39:49Z",   // ISO 8601, UTC, no ms
    "task_id":           <int>,
    "project_id":        <int|null>,
    "project_date_id":   <int|null>,
    "task_label":        "",
    "employee_id":       <int>,
    "is_workplan_event": null,
    "site_id":           null
  }
]
```

The latest event determines current state. Durations are reconstructed by the server from `occured_at` gaps.

### Task ID catalog (from `src/classes/Task.ts`)

| ID | Constant | Meaning |
|---:|---|---|
| 10 | `WORK` / `CLOCKIN` | Regular work time (also represents "clocked in") |
|  8 | `CLOCKOUT` | Ends the shift |
|  5 | `BREAK` | On break |
|  4 | `PROJECT` | Working on a project — **`project_id` is required** |
|  3 | `DRIVE` | Driving time |
|  2 | `LOAD` | Loading time |
|  9 | `DUTY` | Business trip / errand |
|  6 / 7 | `SPECIAL1` / `SPECIAL2` | Custom |
| 11–13 | server-only sentinels (`_NOT_RATED`, `_LOGOUT_DURING_CORE_TIME`, `_ABSENT_DURING_CORE_TIME`) | do not send these from a client |

`clockableTaskIds` = `[10, 5, 4, 3, 2, 9, 6, 7, 8]`. These are the only IDs a client may post.

### Invariants

- `task_id === 4` → `project_id` MUST be present (the `Event` class throws otherwise).
- `occured_at` is UTC ISO 8601, milliseconds suppressed.
- `uuid` is client-generated (v4). Use uuid v4; the server treats it as idempotency key per event.
- Send events in `occured_at` order. The endpoint accepts an array, allowing offline batches.

### Canonical sequence

```
07:30 +00:00  task_id=10                          // clock in
09:00 +00:00  task_id=4,  project_id=1093352      // switch to project
12:00 +00:00  task_id=5                           // break
12:30 +00:00  task_id=10                          // back to work
17:00 +00:00  task_id=8                           // clock out
```

## Suggested MCP tool surface

Keep tools intent-oriented and let the implementation choose the right HTTP call.

### Read tools

```
clockin_status()
  → derived client-side from latest event in latest workday
  → returns { state: "clocked_in" | "on_break" | "on_project" | "driving" | "clocked_out",
              since: ISO, project?: {id, name}, employee_id, ... }

clockin_list_workdays(start?: ISODate, end?: ISODate, employee_id?: number)
  → wraps GET /workdays
  → return Workday[] with nested events; resolve project_id → project.name client-side

clockin_get_totals()
  → GET /timesheets/totals
  → return { flextime_hours, used_vacation_days, max_vacation_days, ... }
     (convert *_seconds → hours for readability)

clockin_list_projects(query?: string)
  → GET /projects, or POST /projects/search if query supplied
  → cache for ~5 min per user
```

### Write tools

```
clockin_clock_in()                                     → event task_id=10
clockin_clock_out()                                    → event task_id=8
clockin_start_break()                                  → event task_id=5
clockin_resume_work()                                  → event task_id=10  (or last main task)
clockin_start_project(project_id, project_date_id?)    → event task_id=4 + project_id
clockin_switch_task(task_id, project_id?)              → generic; validate task_id ∈ clockableTaskIds
clockin_correct_event(event_id, fields)                → PATCH /correction/updateEvent/{eventId}
clockin_delete_event(event_id)                         → DELETE /correction/deleteEvent/{eventId}
clockin_undo(transaction_id)                           → PATCH /correction/undo/{transactionId}
```

Every write helper composes the event payload, including:
- `uuid` (generate fresh v4 per event)
- `occured_at` = now in UTC ISO, ms stripped
- `employee_id` (cached from login or `/device/employee/{id}`)
- `task_id`, optional `project_id`/`project_date_id`

## Effect-TS client outline

Skeleton fitting the existing stack (`@effect/platform` HttpClient, `effect`):

```ts
import { HttpClient, HttpClientRequest, HttpClientResponse } from '@effect/platform'
import { Effect, Layer, Context, Schema, Redacted } from 'effect'

class ClockinConfig extends Context.Tag('ClockinConfig')<
  ClockinConfig,
  { readonly baseUrl: string }
>() {}

class ClockinTokens extends Context.Tag('ClockinTokens')<
  ClockinTokens,
  {
    readonly userToken: Redacted.Redacted<string>
    readonly deviceToken: Redacted.Redacted<string>
  }
>() {}

const NO_AUTH = new Set([
  '/device/upgrade/appv1/appv2', '/auth/login', '/auth/code/login',
  '/auth/user/login', '/auth/user/resetPassword',
  '/registration/company', '/registration/v2/company',
])
const USER_PREFIXES = [
  '/absence', '/company-absence-types', '/correction', '/timesheet',
  '/auth/user/logout', '/auth/user/officeSessionUrl',
  '/auth/user/authorizeQrLogin', '/survey', '/possibleAbsenceSubstitutes',
  '/travelLog/showMany', '/admin/nfc-tags', '/shift-plan', '/admin/',
  '/app-onboarding-tours',
]
const pickToken = (path: string, t: typeof ClockinTokens.Service) => {
  if (NO_AUTH.has(path)) return null
  if (USER_PREFIXES.some(p => path.startsWith(p))) return t.userToken
  return t.deviceToken
}

// Build a client that injects Authorization + device-information per request.
export const ClockinClient = Effect.gen(function*() {
  const cfg = yield* ClockinConfig
  const tokens = yield* ClockinTokens
  const base = yield* HttpClient.HttpClient

  return HttpClient.mapRequest(base, (req) => {
    const path = new URL(req.url).pathname.replace(/^\/v2/, '')  // routing key is sans /v2
    const tok = pickToken(path, tokens)
    let r = HttpClientRequest.setHeader(
      'device-information',
      JSON.stringify({ app_build: '0', app_version: '0.1.0', device_model: 'mcp',
                      operating_system: 'mcp', os_version: '1' }),
    )(req)
    if (tok) r = HttpClientRequest.setHeader('Authorization',
                                             `Bearer ${Redacted.value(tok)}`)(r)
    return r
  })
})
```

Use `Schema` for response decoding once you fill in the placeholder model schemas (extract from real responses or ask the backend team for the source spec).

## Implementation notes & gotchas

1. **52 schemas in the spec are TODO placeholders.** They're flagged with `x-extracted: false` and a description. Fill them in by either:
   - Capturing real API responses during use and inferring shapes,
   - Asking the Clockin backend team for the upstream OpenAPI source,
   - Reading consumer code in the extracted bundle (e.g., `Project.createFromData` reveals Project fields).

   The most important ones to nail down: `Project`, `ProjectArray`, `Employee`, `Workday`/`WorkdayArray`, `Event`/`EventArray` (we have `Event` already), `TimesheetArray`, `TimesheetsTotals`, `AuthToken`/`UserAuthToken`/`DeviceAuthToken`.

2. **`device-information` header is required** on authenticated calls.

3. **Token storage**: this project already encrypts pasted tokens at rest (AES-256-GCM). Store `user_token` and `device_token` as separate columns; both are needed to cover the full surface.

4. **CORS preflight quirk**: the app's webview triggers OPTIONS preflights; an MCP server does not (server-to-server). No special handling required.

5. **Idempotency**: `POST /events/storeMany` uses each event's `uuid` as the idempotency key. Don't reuse uuids on retry — generate fresh ones for each new client intent, but reuse the same uuid if you're retrying the same network call.

6. **Project sync semantics**: the mobile app calls `GET /projects` on each login and replaces its local SQLite table. For an MCP, just fetch on demand with a short TTL cache (5 min is plenty).

7. **Time handling**: send `occured_at` as UTC ISO 8601 with milliseconds suppressed (`...Z`, not `...000Z`). The server is strict.

8. **Error responses**: 401 = bad/missing token, 403 = wrong token tier (sent device_token where user_token expected or vice versa), 422 = validation. Surface these distinctly so the MCP user knows which token to refresh.
